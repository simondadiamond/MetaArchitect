# PATTERN LOG: 2026-03-14 14:00

## 1. THE CORE STATE

**The Goal:** Add rolling multi-week editorial memory to an LLM-assisted content planning pipeline so the planner can make globally coherent decisions, not just locally optimal ones.

**Baseline Belief vs Reality:** Assumed the planner could select good content week-by-week using only the current candidate pool and scoring signals. Reality: without visibility into what has been published recently — which territories, pillars, post classes, and ideas — the LLM is optimizing each week in isolation. It cannot detect territory saturation, trend overload, or idea repetition across weeks because that state simply doesn't exist in its context. The realization was blunt: *how can you expect an LLM to anticipate the future without seeing into the past?*

**The ROI:** Eliminates the need for a human editorial review pass every week to catch "we already covered this territory twice" or "three trend posts in a row." That is a recurring 30–60 minute overhead per week that scales to zero with proper state.

---

## 2. PRODUCTION FAILURE TAXONOMY

**The Symptoms:**
- Planner repeating the same intellectual territory across consecutive weeks with no awareness of saturation
- Trend/product commentary clustering in back-to-back weeks with no signal to break the pattern
- territory_deepening post class never firing because exact slug equality across weeks was near-impossible to achieve (same territory, different angle phrasing at capture time)
- Planner receiving distinct arrays instead of frequency counts — couldn't compute ">60% of recent posts used this pillar" from a set

**The Root Cause:**
Three compounding failures:
1. No rolling state passed to the planner — the LLM was stateless across weeks by design. Each planning session had full candidate context but zero publishing history.
2. Exact string matching for entity identity — territory slugs are generated from free-text angle names at capture time. The same intellectual territory captured on two different days produces two different slugs. Exact equality fails as a cross-week memory mechanism.
3. Sets instead of counts — passing territory_keys_used as a distinct array to an LLM that needs to reason about frequency percentages is a schema mismatch. The LLM cannot derive "this pillar appears 60% of the time" from a deduplicated list.

**The Fix:**
- Added a rolling history fetch step that pulls the last 4 ISO weeks of published post stubs before calling the planner
- Built recentHistory with both distinct arrays (for membership checks) AND frequency count maps (pillar_counts, post_class_counts, territory_key_counts, narrative_role_counts) — the counts are what the planner soft constraint rules actually require
- Replaced exact territory equality with conservative token-overlap matching: split slugs on _, exclude generic domain tokens AND structural suffix vocabulary (error, failure, handling, pattern, detection, strategy), require 3 or more shared meaningful tokens, default to no match when uncertain
- Added aggregateTerritories() to group slug variants under a canonical form so saturation counts are accurate even when the same territory was captured with slightly different phrasing

## 3. SCARCITY METRIC — TIME WASTED

**Time Lost:** ~30–60 minutes per week of editorial review to catch history-blind planner decisions (recurring, not a one-time debug)

**Why it was wasted:** The system was optimizing locally (best posts for this week) without any signal about what had already been published globally (what the audience had seen in the last month).

**If prevented next time:** Rolling state plus count-based soft constraints eliminate the need for a human memory pass entirely. The planner reasons about saturation, trend density, and idea repetition automatically.

## 4. S.T.A.T.E. FRAMEWORK MAPPING

**Structured:** Rolling history object introduced with a defined schema — weeks_sampled, total_posts, distinct arrays, frequency count maps, and prior-week context separated from 4-week aggregates. Type distinction between set-membership checks (arrays) and percentage reasoning (count maps) was explicit in the schema design.

**Traceable:** New post_class, territory_key, source_angle_name, pillar, and thesis_angle fields written to every post stub at creation time — the pipeline now leaves a legible breadcrumb trail that future planning sessions can read back.

**Tolerant:** Proper ISO week boundary handling added (date-based Jan 4 anchor, not a naive week-- if less than 1 set to 52 that breaks on 53-week years). History fetch gracefully returns empty state on first use — no data is treated as no constraint, not a crash condition.

**Explicit:** Conservative territory matching with explicit fallback to no-match. Generic token exclusion list is a named enumerated constant (GENERIC_TERRITORY_TOKENS) — not an implicit heuristic buried in a regex. Post class inference explicitly does NOT classify by narrative role alone (research_commentary does not equal trend_commentary) — classification requires keyword signal from actual content fields.

## 6. CONTENT SEEDS

**Core Insight:** A stateless LLM planner is locally optimal and globally incoherent — it will make the best decision it can with what it sees, and what it sees is always the present. Explicit rolling state is what converts week-by-week decisions into a compounding editorial strategy.
