# PATTERN LOG: 2026-04-03

## 1. THE CORE STATE

**The Goal:** Upgrade a multi-stage AI content pipeline so that a deep research tool runs at intake rather than mid-pipeline, eliminating redundant work and improving output quality from the first stage.

**Baseline Belief vs Reality:** Assumed a two-phase research model (cheap lookup at intake → expensive deep research at planning time) was the right stage boundary design. Reality: the boundary existed because of cost — once the expensive tool became free, the boundary was artificial. The deeper issue: the downstream stage was re-creating a resource the upstream stage had already built and discarded.

**The ROI:** Deep research output (40+ sources, grounded facts) available immediately at intake rather than days later at planning. Downstream research stage reduced from ~5 min to ~30 sec for the common path by reusing the upstream resource instead of re-running the crawl.

---

## 2. PRODUCTION FAILURE TAXONOMY

**The Symptoms:** Pipeline consistently re-running an expensive external operation (40-source web crawl) in a downstream stage — even though an equivalent broad crawl had already been done at intake. The downstream stage was doing targeted refinement on top of redundant duplication.

**The Root Cause:** The upstream stage discarded its resource handle after deriving its output. The downstream stage had no way to reference the upstream work, so it re-created it from scratch. The stage boundary was designed around a constraint (API cost) that no longer existed — but the design was never revisited.

**The Fix:** Store the upstream resource handle as explicit state on the entity record. Downstream stages check for the handle first: if present, run a targeted query against the existing resource (~30 sec); if absent (legacy path or different intake method), run the full operation (~5 min). Two paths, one schema change, no behavioral regression for existing entities.

---

## 3. SCARCITY METRIC — TIME WASTED

**Time Lost:** ~5 min per post stub entering the research stage (full crawl, every time)

**Why it was wasted:** The upstream resource was already built and available — but its ID was never persisted, so it was invisible to downstream stages.

**If prevented next time:** ~4.5 min saved per research run on the common path. At 2x/week posting cadence: ~9 min/week recovered. More importantly: research quality improves at intake, not mid-pipeline — every scored idea immediately has 40-source grounding instead of a single-query overview.

---

## 4. S.T.A.T.E. FRAMEWORK MAPPING

**Structured:** Added `notebook_id` as an explicit state field on the entity record. Upstream stage writes it; downstream stage reads it. The resource handle is now first-class state, not an ephemeral local variable.

**Traceable:** Each NLM call (research_start, research_import, notebook_query) logged with `notebook_id` in `output_summary`. Fast path and full path both produce log entries with distinct `output_summary` labels so execution path is visible in the logs table.

**Auditable:** `provenance_log` on the UIF now records `nlm:{notebook_id}` regardless of which path ran — both paths trace back to the same notebook reference format.

**Tolerant:** Branching logic is explicit: `hasExistingNotebook` boolean derived from `notebook_id ?? null`. If the handle is missing for any reason (legacy entity, failed write), the full crawl runs — no silent degradation.

**Explicit:** Validation gates unchanged. UIF must pass `validateUIF()` before any Airtable write on both paths. Year-anchor gate still enforces query quality.

## 6. CONTENT SEEDS

**Core Insight:** Stage boundaries in AI pipelines are often cost constraints in disguise — when the constraint disappears, the boundary should too; what stays is the resource handle that lets downstream stages reuse upstream work instead of re-creating it.

**ICP Pain:** automation brittleness

**One-line Lesson:** Store upstream resource IDs as explicit state on the entity record — a downstream stage that can't see what the upstream stage built will rebuild it from scratch every time.
